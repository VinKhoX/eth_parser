Complete the eth_parser module implementation in sources/eth_parser.sv.

The design receives an Ethernet byte stream and parses 802.1Q VLAN frames, validates CRC (FCS), and outputs payload bytes to a FIFO based on VLAN ID.

You must implement two submodules:

crc32 – streaming CRC-32 calculator  
payload_extract – header parser, payload buffer, counters, and FIFO output

Instantiate both submodules inside eth_parser.sv. Do not change the ports of the top-level module.


Input Stream

The parser receives a byte stream with the following signals:

rx_valid – asserted when rx_data is valid  
rx_data[7:0] – input byte stream  
rx_err – indicates a receive error during the frame  

Data starts directly at the Destination MAC address (no preamble).

A frame ends when rx_valid goes low for one or more cycles.  
This indicates End Of Packet (EOP).

The last 4 bytes before EOP are the FCS (CRC-32).


Ethernet Frame Format

Destination MAC (6 bytes)  
Source MAC      (6 bytes)  
TPID            (2 bytes) = 0x8100 for VLAN  
TCI             (2 bytes) → VLAN ID is the lower 12 bits  
EtherType       (2 bytes)  
Payload         (variable)  
FCS             (4 bytes)  


VLAN Parsing Rules

Only 802.1Q VLAN frames are accepted.

TPID must equal 0x8100.  
Otherwise the frame is non-VLAN.

Among VLAN frames, only the following VLAN IDs are accepted:

VID = 0  
VID = 1  

All other VLAN IDs must be dropped.


Counter Rules

Counters must be implemented inside payload_extract.

crc_err_cnt  
pkt_err_cnt  
vlan_err_cnt  
valid_pkt_cnt  

crc_err_cnt  
Increment when computed CRC does not match the received FCS.

pkt_err_cnt  
Increment when rx_err occurs during a frame or the frame ends before the full header or payload plus FCS is complete.

vlan_err_cnt  
Increment only when TPID is not equal to 0x8100.

Important rule:  
Frames with TPID = 0x8100 but VLAN ID not equal to 0 or 1 must be silently dropped and must not increment any counter.

valid_pkt_cnt  
Increment when VLAN ID is 0 or 1 and the frame passes CRC and length checks.


Payload Demultiplexing

Payload must be sent to different FIFO outputs depending on VLAN ID.

VID = 0 → fifo_wen[0]  
VID = 1 → fifo_wen[1]

The payload byte should be written to raw_payload.

Payload must not be forwarded immediately.

Instead:

1. Buffer the payload during reception.
2. After CRC is validated at the end of the frame.
3. Replay the buffered payload to the FIFO output.

If CRC fails, discard the buffered payload.


CRC Validation

The last 4 bytes of the frame are the CRC-32 FCS.

CRC must be calculated over all bytes starting from the first Destination MAC byte through the last payload byte.

The final 4 FCS bytes must NOT be included in the CRC calculation.

After the frame ends, compare the computed CRC with the received FCS.


Frame Boundary Rule

A frame starts on the first cycle where rx_valid becomes high after being low.

A frame ends when rx_valid becomes low.

When rx_valid goes low, treat the previous 4 bytes as the FCS.

After CRC validation:
- If CRC is correct and VLAN ID is valid, replay the buffered payload to the FIFO.
- Otherwise discard the buffered payload.


First Byte Alignment

The first byte of the frame must not be skipped.

The first cycle where rx_valid becomes high after idle corresponds to the first Destination MAC byte.

That byte must be included in CRC and in header parsing.

If a registered start-of-packet signal is used, capture the first byte so the stream is not shifted by one cycle.


Submodule Interfaces

payload_extract must expose signals for the CRC engine:

crc_init  
crc_data  
crc_vld  
crc_computed  

crc_init – pulse at start of frame  
crc_data – byte sent to CRC module  
crc_vld – indicates crc_data is valid  
crc_computed – CRC result from crc32  


CRC Module

Implement a streaming crc32 module with the following interface:

input clk  
input rst  
input init  
input [7:0] data_in  
input data_vld  
output [31:0] crc_out  

init resets the CRC calculation at the start of a frame.


Payload Buffer

payload_extract must include an internal payload buffer.

Use parameter:

parameter MAX_PKT = 2048

The buffer stores payload bytes until CRC validation completes.


Top-Level Responsibilities

eth_parser.sv must instantiate crc32 and payload_extract and connect them together.

The top-level module must not implement parsing or counters directly.

All parsing and statistics logic must be implemented inside payload_extract.


File Organization

Implement the following files:

sources/eth_parser.sv  
sources/crc32.sv  
sources/payload_extract.sv  

The design must compile and pass the Cocotb tests in:

tests/test_eth_parser.py


Hints

Use a finite state machine to parse the frame header.

You may track a byte counter starting from the first byte of the frame.

Example byte positions:

0-5   Destination MAC  
6-11  Source MAC  
12-13 TPID  
14-15 TCI  

VLAN ID is the lower 12 bits of TCI.

Do not include the final 4 FCS bytes in the CRC calculation.

Buffer payload until CRC validation is complete.

Replay payload only after CRC passes.

Reset internal state when rx_valid goes low.
